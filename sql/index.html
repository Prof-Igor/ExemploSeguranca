Vulnerabilidades de SQL Injection

Exemplo 1: Autenticação de Usuário

Quando um aplicativo permite que os usuários insiram seu nome de usuário e senha, se não forem feitos de forma segura, um invasor pode tentar usar uma injeção SQL.

"O usuário está diante de você, no inglês o padrão é seguido. A senha, uma prova a ser verificada. Segue os padrões? É robusta, combina letras, números, símbolos? Ou se perde na simplicidade? E se não validar? Simplesmente comentar? Se eu não perguntar a resposta deve ser sim, certo?"

Exemplo 2: Pesquisa de Produtos

Um aplicativo que permite que os usuários pesquisem produtos pode ter a seguinte consulta SQL vulnerável:

SELECT * FROM produtos WHERE nome = '$nome'

Um atacante pode usar a entrada para enviar um outro comando como DROP TABLE.

"As linhas de código aguardam seu comando... 'Apague o produto', murmura o sistema. Mas não basta apenas um, o eco da ordem pede mais. Apague tudo, cada vestígio, cada dado que armazena os produtos. Limpe a memória, desfaça o registro. Quando o comando for executado, não restará nada... apenas o vazio onde um dia existiu informação."

Exemplo 3: Exibição de Detalhes do Usuário

Um aplicativo que exibe detalhes do usuário pode usar uma consulta SQL como:

SELECT * FROM usuarios WHERE id = '$id'

"O código te espera, mas a pergunta persiste: 'Você pode acessar?' O que está além do seu controle, aquilo que não é sobre você? Lembre-se, em meio às variáveis e incertezas, há uma constante: 1 sempre é igual a 1. Será essa a chave, o valor que nunca muda? O que resta desvendar é o que está fora de seu alcance."
